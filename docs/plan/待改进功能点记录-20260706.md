## IMP-001 zip 导入前自动剥离冗余包与缓存

| 字段 | 内容 |
| --- | --- |
| 状态 | 已完成（DEV-034） |
| 优先级 | P1 |
| 背景 | 用户打包 zip 时常带入 `node_modules/`、`__pycache__/`、`.venv/`、`__MACOSX/` 等；其中符号链接会触发 zip 安全审计拒绝导入（如 20260706 声纹 V3 演示包）。不应要求用户重新打包，而应在导入链路自动清理后再扫描、构建、托管。 |
| 目标 | 导入前规范化 zip 内容：剔除可重建依赖目录与平台 junk；保留源码、`package-lock.json` / `requirements.txt`、`dist/`（若有）；清理后再走 scanner → build → start。 |
| 非目标 | 不把 `node_modules/`、`.venv/` 等依赖目录作为可恢复数据保存；不尝试修复损坏 zip；不放宽业务源码目录中的恶意 symlink / zip slip 拦截。 |
| 清理规则 | 默认剥离任意层级的 `node_modules/`、`__pycache__/`、`.pytest_cache/`、`.mypy_cache/`、`.ruff_cache/`、`.venv/`、`venv/`、`.git/`、`.DS_Store`、`__MACOSX/`；保留 lockfile、源码、`public/`、`dist/`、`build/`、配置文件。**不剥离裸 `env/`** —— 应用常用该目录存放环境配置/示例，与「保留配置文件」规则冲突；Python 虚拟环境由 `.venv/`、`venv/` 覆盖。 |
| 安全边界 | 先按 zip 成员路径做「可剥离前缀」分类，被剥离成员**不落盘**；对保留成员继续执行 `audit_zip_members` 和 `_safe_extract` 的 zip slip / symlink 防御。也就是说，`node_modules/.bin/*` symlink 可被剥离通过，`src/evil -> /etc/passwd` 仍拒绝。 |
| 实现触点 | **Python**：`importer.py` 增加 zip sanitization 阶段，可抽成 `sanitize_zip_members` 便于测试；`security.audit_zip_members` 不需要全局降级，只对已剥离成员跳过审计。**CLI**：`lwa import` 输出剥离摘要（目录、文件数、体积变化、是否含 symlink）。**Skill**：新增或扩展 `lwa-import-zip` 类 skill，文档化「可带 node_modules，lwa 会自动清理」。 |
| 验收 | 含 `node_modules/.bin` symlink 的原版 zip 可一键 import+start；剥离日志可审计；保留 `package-lock.json` 后前端仍可构建；安全回归（zip slip / 源码目录恶意 symlink / Windows 盘符路径）仍拒绝；重复 import 同一 zip 的 hash/实例命名行为保持稳定。 |

## IMP-005 Caddy 简易访问频率限制（内网防护）

| 字段 | 内容 |
| --- | --- |
| 状态 | 已完成（DEV-035） |
| 优先级 | P2 |
| 背景 | 公司内网使用，需防误操作或简单滥用，不需要完整 WAF。 |
| 目标 | Caddy 模式下对静态站点启用 rate limit：**约 3 次/秒/客户端**（可配置）；builtin 模式可文档说明暂不支持或在反向代理层补充。 |
| 范围 | 仅保护实例静态访问入口；管理页 API 的鉴权/限流另列议题。健康检查、本机访问是否豁免需要在实现前明确，默认不豁免更简单。 |
| 参考 | Caddy `rate_limit` 插件或 `http.handlers.rate_limit`（需评估是否引入自定义 Caddy 构建依赖）；阈值写入 `local-web.yml` 如 `staticRateLimitRps: 3`，默认关闭或仅在 Caddy 能力可用时开启。 |
| 风险 | 若依赖非默认插件，用户本机 Caddy 可能不支持对应 directive，导致 reload 失败；实现必须先做能力检测或提供明确 fallback，不能让所有静态站点因限流配置不可用而下线。 |
| 配置草案 | `staticRateLimit: { enabled: false, rps: 3, burst: 6 }`；生成 Caddy 配置前校验 `enabled`、`rps`、`burst` 范围。 |
| 验收 | Caddy 支持限流时超阈值返回 429 或等价拒绝；插件/handler 不可用时给 WARN 并保持站点可访问；builtin 模式输出「暂不支持」说明；配置 reload 失败可回滚旧 Caddyfile。 |

## IMP-006 路径别名路由（下一阶段，可选）

| 字段 | 内容 |
| --- | --- |
| 状态 | 已完成（DEV-036） |
| 优先级 | P3 |
| 背景 | 每实例独立端口难记；用户希望 `http://<LAN-IP>/voiceprint-app-demo/` 访问，而非记 `18001`。 |
| 规则 | **默认行为不变**：用户未指定路径别名、且 CLI/Skill 未传 `--path-alias`（或等价参数）时，仍仅通过 **hostPort**（如 `:18001`）访问。**仅当用户主动指定**路径别名时，才在统一入口（Caddy 反向代理）上注册 `/<path-alias>/` → 该实例 upstream 的映射。 |
| CLI/Skill | `lwa import inbox/foo.zip --path-alias voiceprint-app-demo`（可选）；manifest 写入 `pathAlias`；`lwa start` 后同时输出端口 URL 与路径 URL（有别名时）。Skill 文档说明：不传别名则只用端口，传了才启用路径访问。 |
| 校验 | slug 格式（小写、数字、连字符）；与 `instance_id` 可相同可不同；全局唯一、不可与 reserved 路径冲突（如 `/api`）。 |
| URL 规则 | 统一规范为带尾斜杠路径：`/<alias>/`；访问 `/<alias>` 可 301 到 `/<alias>/`；内部转发优先用「去前缀」语义，避免应用收到 `/alias/...` 后找不到资源。 |
| SPA 注意 | 许多前端构建产物会写死 `/assets/...` 绝对路径，路径别名会导致浏览器请求根路径资源。实现前需选择策略：仅支持相对路径/已配置 base 的产物；或在 build 阶段注入 base；或文档明确限制。不能默认保证所有 SPA 在子路径下可用。 |
| 数据与清理 | `pathAlias` 写入 `local-web.json` 与 registry 索引；start/restart 生成路由；stop 可保留别名登记但禁用 upstream；remove 必须清理别名路由；重命名/改别名需处理冲突和旧路由删除。 |
| 依赖 | 优先 Caddy 模式；builtin 多端口模式需评估是否合并为单入口 + 路由。 |
| 示例 | 声纹演示：用户指定 `voiceprint-app-demo` → `http://10.x.x.x/voiceprint-app-demo/` 与 `:18001` 并存。 |
| 验收 | 不传别名时输出与 V1 完全一致；传别名后端口 URL 与路径 URL 同时可用；别名冲突/保留字返回明确错误；remove 后旧路径 404；含绝对资源路径的 SPA 有明确限制提示或测试覆盖。 |

| 目标 | 实例详情/列表展示：`原端口 --→ hostPort`（例：`33001 → 18000`）；数据来源：`local-web.json` 的 `internalPort` / manifest `network.internalPort` / scanner 识别结果。 |

| UI 草案 | 端口列主显示可访问端口：`18000`；副信息显示 `33001 → 18000`；鼠标悬浮解释「应用内部端口 → 宿主端口」。移动窄屏可折叠为 `33001→18000`。 |
| 验收 | Node/Python 容器显示内部端口映射；静态实例不显示误导性「80 → hostPort」；前端列表、详情页与 `lwa status/list` 口径一致；缺失 internalPort 时不报错、不显示空白。 |

## IMP-008 源代码更新与工作区热重载（`lwa update`）

| 字段 | 内容 |
| --- | --- |
| 状态 | 已完成（DEV-038） |
| 优先级 | P1 |
| 背景 | V1 已有 `lwa setup`（宿主机工具检测）与 `lwa init`（工作区初始化），但**缺少「lwa 自身升级 + Runtime 重载」**闭环。开发者在仓库 `git pull` / 改代码后，`pip install -e .` 虽会更新 Python 包，**已在跑的管理页/daemon 子进程仍持有旧代码**；工作区 `skills/` 也不会自动与包内 SKILL 同步。 |
| 目标 | 一条命令（或 Skill 引导）完成：**刷新安装 → 同步工作区附属物 → 重启 lwa 自有服务 → 可选校验**；用户导入的实例（apps/）默认**不**重建，除非显式 `--restart-instances` 或变更涉及网关/托管逻辑。 |
| 与 setup/init 关系 | `setup` = 机器环境；`init` = 首次建工作区；**`update` = 已有工作区 + 新版本 lwa 代码**。 |

### 规划命令形态

```bash
# 在工作区目录（runtime/）或仓库根目录均可调用
lwa update [OPTIONS]

选项（草案）：
  --workspace, -w PATH     工作区根（默认识别 local-web.yml）
  --repo PATH              lwa 源码根（默认识别 editable 安装路径或当前 git 根）
  --dry-run                仅做识别与计划展示，不执行 pip/同步/重启
  --skip-pip               跳过 pip install -e .（已手动装过）
  --sync-skills            同步包内 skills/ → 工作区 skills/（默认开）
  --sync-templates         同步 templates/（默认关，避免覆盖用户改过的模板）
  --restart-manager        重启管理页后台（默认开，若 managerEnabled）
  --restart-daemon         若 daemon 在跑则重启（默认开）
  --restart-instances      对所有 running 实例 restart（默认关，耗时长）
  --doctor / --no-doctor   结束后是否跑 lwa doctor（默认开）
  --json                   机器可读摘要
```

### 执行步骤（实现时）

| 步骤 | 动作 | 说明 |
| --- | --- | --- |
| 1 | 识别上下文 | 定位 workspace（`local-web.yml`）与 repo（editable 安装路径 / 当前 git 根 / `--repo`）；无法定位时给可执行错误提示 |
| 2 | 预检与 dry-run | 读取当前版本、manager/daemon 运行态、running 实例列表、将要同步的 skills/templates；`--dry-run` 到此结束 |
| 3 | `pip install -e .` | 在源码根执行，刷新 editable 安装与依赖；非 editable/wheel 安装场景需提示用户传 `--repo` 或跳过 pip |
| 4 | 同步 skills | 复用 `init_workspace._copy_default_skills`（新增文件覆盖、不删用户自定义）；输出新增/覆盖/跳过清单 |
| 5 | 配置迁移（可选） | 若 `local-web.yml` 缺新字段（如 `managerEnabled`），合并默认值并提示 diff；破坏性迁移必须先备份 |
| 6 | 重启 manager | 仅在原本启用或运行时重启：`manager off` → `manager on`；失败时保留错误摘要和旧 PID/日志路径 |
| 7 | 重启 daemon | 若 daemon 原本 running：`daemon off` → `daemon on`；原本 stopped 则不自动开启 |
| 8 | 实例策略 | **默认不动** running 实例；`--restart-instances` 时跳过 building/queued/pending，逐个 `lwa restart <id>` 并汇总失败 |
| 9 | 校验 | `lwa doctor`；输出新 manager URL / token 提示；`--json` 时同样保留人可读日志路径 |

### 为何不默认重启所有实例

- 静态/容器实例进程与 **lwa CLI 代码** 解耦：多数 bugfix 只影响管理页、import、doctor。
- 涉及 `hosting` / `static_gateway` / `manager_api` 行为变更时，文档建议 `lwa update --restart-instances` 或手动 `lwa restart <id>`。
- 避免误中断长时构建（building 实例）。

### 失败处理与安全边界

- 任一步失败时返回非零退出码，并在摘要中标记已完成/未完成步骤；不要吞掉 `pip`、manager、daemon 的原始错误。
- `pip install -e .` 成功但 manager 重启失败时，不回滚 Python 包；提示用户查看 `run/manager.json`、`logs/` 并可手动 `lwa manager on`。
- `--sync-templates` 默认关闭；开启时只覆盖由 lwa 管理且未被用户改过的模板，或先生成 `.bak`。
- 配置迁移和 registry/schema 迁移分开处理；IMP-008 首版只做缺省字段补齐，不做破坏性 schema 迁移。
- `--restart-instances` 对每个实例独立失败，不因单个实例失败中断后续实例，但最终退出码应反映存在失败。

### Skill 触点

- 新增 **`lwa-update-runtime`**（见 `src/local_webpage_access/skills/lwa-update-runtime/SKILL.md`）：AI 助手在用户说「代码更新了怎么刷 runtime」时走同一流程；在 `lwa update` 未实现前给出**手动清单**。

### 验收

- `git pull` + `lwa update` 后管理页版本号与 `lwa version` 一致（如 V0.3.2 → V0.4.0）。
- 工作区 `skills/` 含新 SKILL 文件；registry/apps 数据不丢。

### 当前临时做法（实现前）

在仓库根与工作区分别执行：

```bash
# 1. 刷新 Python 包
cd /path/to/local-webpage-access && pip install -e .

# 2. 重启 lwa 自有服务（在 runtime 工作区）
cd runtime
lwa manager off && lwa manager on
# 若曾开启 daemon：
lwa daemon off && lwa daemon on

# 3. 可选：托管逻辑变更后重启业务实例
lwa restart voiceprint-v3-demo

# 4. 校验
lwa doctor
lwa version
```

## IMP-009 实例 zip 包更新（再导入识别与原地升级）

| 字段 | 内容 |
| --- | --- |
| 状态 | 已完成（DEV-039） |
| 优先级 | P1 |
| 背景 | 用户常先 import 一个 zip 并 start 跑起来，之后把**同一项目的较新版本 zip** 再放进 `inbox/` 或交给助手导入。V1 行为：`import_zip` 始终走**新建实例**路径（`_resolve_unique_id` 在 slug 冲突时生成 `demo-2`、`demo-3`…），**不会**识别「这是对已有实例的内容更新」。结果：端口/hostPort 变了、旧实例仍占资源、用户需手动 stop/remove 旧实例，体验割裂。 |
| 与 IMP-008 区分 | **IMP-008** = lwa **工具本身**升级（`pip install -e .`、重启 manager/daemon）；**IMP-009** = 用户**业务 zip 包**升级（替换 `apps/<id>/current`、可选重建、重启**同一 instance_id**）。二者独立，可组合使用（先 `lwa update` 再 `lwa import --update`）。 |
| 目标 | 支持在**保留 instance_id、hostPort、data/ 持久数据、desiredState** 的前提下，用新 zip 覆盖业务源码并触发 rebuild/restart，让用户感知为「同一个网页更新了」，而不是「又多了一个新实例」。 |

### 识别策略（自动 + 显式，多层兜底）

| 层级 | 方式 | 说明 |
| --- | --- | --- |
| **显式（优先）** | `lwa import inbox/foo.zip --update <instance-id>` | 用户或 Skill 明确指定更新目标；最可靠，冲突时以此为准。 |
| **显式（别名）** | `lwa import inbox/foo.zip --name voiceprint-v3-demo` 且 registry 已有同 id | 若 id 已存在：V1  today 会建 `-2`；IMP-009 改为**提示**「是否更新已有实例」或要求加 `--update` / `--replace` 旗标，避免 silent 新建。 |
| **半自动** | hash / 元数据比对 | 新 zip 的 sha256 与 manifest `sourceZipHash` **相同** → 跳过或 WARN「包未变化」；**不同**且 inbox 文件名 stem / `--name` slug 与已有 `instance_id` 匹配 → 建议更新该实例（CLI 交互或 `--yes` 非交互）。 |
| **弱信号（可选）** | 包内 `package.json` version、`local-web.json` 内嵌标记、zip 内 README 项目名 | 仅作**建议**，不能单独作为自动覆盖依据（易误判）；可输出「疑似更新 voiceprint-v3-demo，请确认」。 |
| **daemon inbox** | watcher 发现同名 stem 的 zip | 默认**不**自动覆盖 running 实例；写事件 + 可选「待更新」队列，或要求用户/manage 页确认（防误覆盖生产演示）。 |

**V1 已有基础**：manifest 已存 `sourceZipHash`、`sourceZipPath`；registry `instances` + `apps/<id>/source/original.zip` 可对比。缺的是「更新路径」而非元数据。

### 非目标

- 不从远程 URL / Git 拉取更新（仅本地 zip / inbox 文件）。
- 不自动 merge 用户改过的 `apps/<id>/current` 与 zip  diff（更新 = 以 zip 为准覆盖 `current/`，`data/` 默认保留）。
- 不在未确认时 silent 删除旧实例或自动 remove `demo-2` 等历史误导入实例。
- runtime 从 static 切到 docker-compose（或反向）的**形态迁移**不在首版承诺；检测到 kind/runtime 变化时应拒绝或走「新建实例」并明确提示。

### 规划命令形态

```bash
# 显式更新（推荐）
lwa import inbox/V3-B.zip --update voiceprint-v3-demo

# 等价别名（草案，实现时二选一）
lwa reimport voiceprint-v3-demo inbox/V3-B.zip
lwa update-instance voiceprint-v3-demo --from inbox/V3-B.zip

选项（草案）：
  --update, -u INSTANCE_ID   更新已有实例（原地），而非新建
  --replace                  与 --update 联用：允许覆盖 current/（默认即覆盖）
  --no-restart               仅换包+构建，不自动 start（适合维护窗口）
  --restart                  更新完成后 restart（默认：若 desiredState=running 则 restart）
  --keep-data                保留 data/（默认开）
  --yes / -y                 非交互确认（daemon/CI）
  --dry-run                  展示将替换的文件/hash 差异、是否需 rebuild，不写盘
```

### 执行步骤（实现时）

| 步骤 | 动作 | 说明 |
| --- | --- | --- |
| 1 | 校验目标实例存在 | 不存在则报错并建议普通 `import` |
| 2 | 计算新 zip hash | 与 `sourceZipHash` 比较；相同可跳过（exit 0 + 提示） |
| 3 | Sanitize + 解压 | 复用 IMP-001 剥离规则；备份旧 `original.zip` → `source/original.zip.bak` 或按版本命名 |
| 4 | 覆盖 `current/` | 清空后解压新内容；**不**动 `data/`、`public/` 在 rebuild 前由 hosting 重写 |
| 5 | 重新 scanner | kind/runtime 变化 → 拒绝或需 `--force-kind-change`（首版建议拒绝） |
| 6 | 更新 manifest | `sourceZipHash`、`updatedAt`、必要时 `version` 字段；保留 `network.hostPort`、instance_id |
| 7 | 构建与托管 | 若实例原 running：`stop`（或仅停网关）→ `rebuild_instance` / `host_instance` → 恢复 desiredState |
| 8 | 事件与 registry | `import`/`update` 事件写入 diff 摘要（旧 hash→新 hash、是否 rebuild） |

### 管理页触点（下一阶段）

- 实例详情或操作区：**「更新包…」**（上传/选择 inbox 内 zip 或填路径 — V1 可先只做「填 inbox 路径 + 选实例」API）。
- `POST /api/instances/{id}/update`（multipart 或 `{ "zipPath": "inbox/foo.zip" }`），复用与 CLI 同一套 Python 逻辑。
- 列表可展示 `sourceZipHash` 前 8 位或 manifest `version`，便于用户确认是否已更新。

### Skill 触点

- 扩展 **`lwa-import-zip`**：文档说明「同一项目新版本 → `--update <id>`」；助手收到新 zip 时先查 registry 是否已有同 slug / 用户指名的实例，**询问或带 `--update`**，避免无脑 `import` 出 `-2`。
- 与 IMP-001 联动：更新路径同样走 zip sanitization。

### 失败处理

- 解压/扫描/build 任一步失败：**回滚**到更新前 `current/` + `sourceZipHash`（或保留 `.bak` 直至成功）；实例 status 置 `failed` + `last_error`。
- 更新过程中持 `instance_lock`，与 start/stop/rebuild 互斥。
- 端口/hostPort **不变**；避免更新后 LAN URL 变化。

### 验收

- 已有 `voiceprint-v3-demo` running 于 18001；导入同项目新 zip `--update voiceprint-v3-demo` 后：**仍为同一 id、同一端口**，HTTP 200，内容为新版。
- 相同 hash 再次 `--update`：明确提示「包未变化」，无 rebuild。
- 不传 `--update` 且 slug 冲突：不 silent 创建 `-2`（交互提示或错误 + 建议 `--update`）。
- `data/` 内 SQLite/上传文件在更新后仍在（全栈/SQLite 样例）。
- 管理页/API 与 CLI 更新结果一致；事件链可审计（旧 hash → 新 hash）。
- 单测：mock zip，覆盖 hash 相同跳过、kind 变化拒绝、失败回滚、保留 hostPort。
